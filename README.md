### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут управлять отоплением в доме — включать и выключать отопление в помещениях.
- Система поддерживает добавление нового датчика/реле специалистом компании, который выезжает к пользователю для
  подключения системы отопления в доме.

**Мониторинг температуры:**

- Пользователи могут просматривать температуру в выбранной комнате и с конкретного датчика.

**Подключение устройств**

- Подключение датчиков и умных устройств доступно только специалистам компании

### 2. Анализ архитектуры монолитного приложения

#### Доступность

Система монолитная с одной базой данных, а значит есть единая точка отказа. Обновление требует остановки всего
приложения, а значит есть плановые простои. Взаимодействие синхронное, поэтому при проблемах с сетью могут возникать
зависания и система может перестать отвечать. Таким образом, доступность системы можно оценить как слабую.

#### Масштабируемость

Из-за монолитной архитектуры масштабирование возможно только вертикальное масштабирование. При росте числа устройств
нагрузка на систему будет расти почти линейно, так как запросы синхронные. Все функции приложения конкурируют за
ресурсы одного монолитного сервера. Таким образом, архитектура почти не масштабируется под To-Be состояние.

#### Изменяемость

В таблице sensors хранятся все типы устройств, что усложняет добавление новых типов устройств. Так как вся логика
собрана в одном монолите изменения одной части системы может привести к поломкам в других частях. Таким образом,
изменяемость можно оценить как слабую.

#### Наблюдаемость

В базе данных хранятся только последние значения для датчиков, соответственно, нет истории показаний. Из-за этого
сложно определять, как и когда начались проблемы. Нет явной телеметрии самой системы — метрик и логов. Таким образом,
наблюдаемость системы тоже можно оценить как слабую.

#### Сильные стороны As-Is архитектуры

- Понятность. Один сервис и одна база данных до определенного момента упрощает понимание системы, что ускоряет
  онбординг для небольшой команды.
- Простота развертывания. Для развертывания достаточно одного пайплайна без координации релизов нескольких сервисов.
- Целостность данных. Так как данные хранятся в одной схеме PostgreSQL, можно использовать транзакции, что упрощает
  обеспечение согласованности данных.

### 3. Определение доменов и границы контекстов

#### Домены

1. Управление отоплением.
2. Сервисное обслуживание.
3. Управление пользователями.

Домены «Сервисное обслуживание» и «Управление пользователями» в системе не рассматриваются, поэтому поддомены
выделялись только для домена «Управление отоплением».

#### Поддомены домена «Управление отоплением»

1. Мониторинг температуры.
2. Установка и отключение устройств.
3. Управление отоплением.

Домены и поддомены выделялись без учета кода существующего приложения, а только на основе описания текущего
состояния продукта и компании, так как с точки зрения DDD домены и поддомены «выявляются», то есть являются
производными бизнес-процессов. При реальной работе над выявлением доменов и поддоменов необходимо было бы активно
консультироваться с представителями бизнеса.

На основании имеющихся данных невозможно однозначно понять, как были определены ограниченные контексты в текущем
решении. Основной вопрос — как осуществляется включение/выключение отопления пользователем.

#### Контексты

Контексты определялись исходя из существующей системы без оглядки на качество такого выделения.

Контексты:

1. Smart Home Sensor Management. Реализован в монолитном приложении. В этом контексте содержится вся логика системы.
2. Heating Management. Содержится в монолитном приложении, отсутствует в приведенном коде. Отвечает за включение и
   выключение отопление через умные реле. Явно нигде не указана, но без этой системы не ясно, как осуществляется
   включение/выключение отопления пользователями. Сделано предположение, что эта система считывает данные из базы данных
   Smart Home Sensor Management и взаимодействует с нужными реле.
3. Smart Devices. Внешняя аппаратная система. Сами устройства — датчики и реле. В явном виде в коде не присутствуют, но
   однозначно присутствуют в системе.

Последние 2 поддомена в коде не упоминаются, но должны существовать для функционирования системы при обозначенных
выше предположениях.

# Задание 2. Проектирование микросервисной архитектуры

Бизнес-процессы, необходимые для To Be состояния системы существенно отличаются от состояния As Is, поэтому процесс
проектирования лучше начать с начала, то есть с анализа требований и выявления доменов.

Для нового состояния системы получен следующий набор доменов:

1. Управление пользователями.
2. Поддержка пользователей.
3. Продажи и маркетинг.
4. Сайт компании. Личный кабинет пользователя, возможные интеграции с поставщиками оборудования, новости компании и
   т. д.
5. Back-office. С учетом существенного увеличения числа бизнес-доменов в целевом состоянии целесообразно выделить этот
   домен.
6. Видеонаблюдение. Работа с видео кардинально отличается по нефункциональным требованиям от работы с другими
   устройствами умного дома, поэтому ее лучше вынести в отдельный домен. Кроме того, сейчас неясно, как будет
   организована работа с этим доменом — собственная инфраструктура, готовые решения или какой-то гибридный вариант.
7. Совместимые устройства. Каталог поддерживаемых устройств и их возможностей, поддерживаемы протоколы, прошивки и
   обновления совместимых устройств.
8. Платформа умного дома. Непосредственная работа с умными устройствами и инфраструктура для их функционирования в
   системе.
9. Функционирование умного дома. Доступные клиентам сценарии работы с умным домом.

Домены 1-6 для текущей задачи целесообразно рассматривать как лежащие за границами проектируемой системы, так как
эти домены очень сильно завязаны на конкретные бизнес-процессы и не могут быть корректно спроектированы без выяснения
деталей. Поэтому в дальнейшей работе я сконцентрировался на доменах 7-9.

Поддомены домена «Совместимые устройства»:

1. Каталог поддерживаемых устройств. Список устройств, которые могут быть подключены к системе. Формируется исходя
   из данных, предоставляемых компаниями-партнерами. Используется для подключения только тех устройств, с которыми
   умеет работать система.
2. Поддерживаемые протоколы. Список поддерживаемых протоколов для различных устройств и вспомогательные инструменты
   для работы с этими протоколами.
3. Прошивки и обновления устройств. Реестр доступных прошивок и обновлений для поддерживаемых устройств.
4. Доступные возможности устройств. Каталог поддерживаемых устройств с фокусом на их характеристиках и возможностях с
   учетом версий прошивок.

Поддомены домена «Платформа умного дома»:

1. Реестр устройств. Реестр устройств, подключенных к системе.
2. Реестр помещений. Реестр помещений, в которых установлены устройства в системе. Поддерживает топологию устройств
   с учетом хабов и физических помещений.
3. Жизненный цикл устройств. Подключение устройств к системе, их обновление и отключение.
4. Маршрутизация команд. Шлюз между системами управления умным домом и платформой.
5. Телеметрия. Сбор и хранение данных с устройств для поддержки функционирования системы, аналитики и устранения
   неполадок.
6. Диагностика. Инструменты взаимодействия с устройствами для оценки их состояния и работоспособности.

Поддомены домена «Функционирование умного дома»:

1. Управление умным домом. Инструменты работы пользователей для добавления помещений и устройств в систему.
2. Поддерживаемые возможности устройств. Функции, доступные для подключенных пользователями устройств, а так же для
   групп устройств и помещений.
3. Управление сценариями. Инструменты создания сценариев работы с умным домом для пользователей. Включает как
   готовые сценарии, так и инструменты создания кастомных сценариев из поддерживаемых функций устройств.

Полноценно реализовать систему для поддержки всех выявленных поддоменов с имеющейся командой в разумные сроки вряд
ли возможно, поэтому целесообразно выделить несколько сервисов, чтобы создать PoC-решение с использованием
микросервисной архитектуры без резкого отказа от существующего монолита. Для этого решено использовать решение,

# Задание 4. Создание и документирование API

### 1. Тип API

В системе используются два типа API:

1. **REST / HTTP API (OpenAPI)**

   Применяется для синхронного взаимодействия:
   - веб/мобильный клиент <-> Devices service;
   - Devices service <-> Telemetry service;
   - монолит <-> эмулятор temperature‑api;
   - Devices service <-> монолит (регистрация устройств отопления).

   Причины выбора:
   - HTTP+JSON де‑факто стандарт для клиентских приложений и интеграций;
   - существующий монолит уже предоставляет REST API, поэтому для новых сервисов логично использовать тот же подход;
   - в условиях создания MVP, в котором нет специфических нефункциональных требований, логично минимизировать число
     используемых протоколов.

2. **Сообщения через Kafka (AsyncAPI)**

   Используются для асинхронного обмена телеметрией:
   - сервис Temperature service публикует события с показаниями температуры в Kafka;
   - Telemetry service подписывается на топик и записывает измерения в Telemetry DB.

   Причины выбора:
   - поток измерений от устройств естественно моделируется как асинхронный событийный поток, а не как синхронные
     REST‑запросы;
   - AsyncAPI позволяет описать формат сообщений и топика, чтобы задокументировать контракт между producer'ом и
     consumer'ом.

# Задание 5. Работа с docker и docker compose

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

1. сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе
   /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```

2. Приложение следует упаковать в Docker и добавить в docker compose. Порт по умолчанию должен быть 8081

3. Кроме того для smart_home приложения требуется база данных - добавьте в docker compose файл настройки для запуска
   postgres с указанием скрипта инициализации ./smart_home/init.sql

Для проверки можно использовать Postman коллекцию smarthome-api.postman_collection.json и вызвать:

- Create Sensor
- Get All Sensors

Должно при каждом вызове отображаться разное значение температуры

Ревьюер будет проверять точно так же.
